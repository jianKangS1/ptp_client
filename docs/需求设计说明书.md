## 1. 引言

### 1.1 编写目的

本文档描述「时间同步多协议客户端」软件的需求与设计要点，作为开发、测试、验收及与外部系统集成的依据。读者包括研发、测试、运维及项目管理相关人员。

### 1.2 项目背景与目标

在运营商、专网、工业等场景中，设备需要同时或分别通过 **IEEE 1588 PTP**、**1588 ACR（自适应时钟恢复）** 与 **NTP** 与上游时间源保持同步或恢复时钟。本软件在一套可部署的后端（默认 Python）中提供上述三类客户端能力，并支持**多个客户端实例**与各自对端 **Server/Grandmaster** 的并行交互，满足多域、多接口、多业务隔离的需求。

### 1.3 文档范围

- **范围内**：三类客户端的逻辑行为、多实例模型、配置与观测、与 Server 的交互边界、非功能需求概要。  
- **范围外**（除非后续版本纳入）：硬件驱动与网卡固件细节、完整电信网管北向接口、具体芯片 PHC 驱动选型。

### 1.4 术语与缩写

| 术语 | 说明 |
|------|------|
| PTP | Precision Time Protocol，IEEE 1588 精确时间协议 |
| GM | Grandmaster，PTP 主时钟 |
| BC / TC | Boundary Clock / Transparent Clock |
| ACR | Adaptive Clock Recovery，基于报文时间信息的自适应时钟恢复 |
| NTP | Network Time Protocol |
| E2E / P2P | End-to-End / Peer-to-Peer 延迟测量机制 |
| BMC | Best Master Clock Algorithm，最佳主时钟算法 |

### 1.5 参考标准与资料（示例）

- IEEE 1588-2008 / IEEE 1588-2019  
- RFC 5905（NTPv4）及相关 SNTP 资料  
- ITU-T G.8275.1 / G.8275.2 等（若启用电信配置文件，在配置中声明）  
- 项目内部接口规范（REST/gRPC 等，待版本化定稿）

## 2. 系统概述

### 2.1 产品定位

本软件为**时间同步客户端中间件/服务**：向上游时间服务器发起或响应协议交互，完成偏移与延迟估计、频率/相位跟踪（含 ACR 路径），并向本机或上层应用暴露同步状态与时间质量信息。

### 2.2 用户角色

| 角色 | 描述 |
|------|------|
| 运维/集成方 | 配置多实例、网络接口、对端地址、观测告警 |
| 上层应用 | 通过 API 或回调读取时间、状态、质量等级 |
| 时间源设备 | PTP GM、NTP Server 等，作为对端 |

### 2.3 运行环境（待配置项确认）

- **操作系统**：Linux（推荐，便于 PHC / `SO_TIMESTAMPING`）；Windows 为可选目标，需在详细设计中说明能力降级项。  
- **运行时**：Python 3.11+（与项目规则一致）。  
- **网络**：二层组播或 UDP/单播，依协议与配置而定。

## 3. 功能需求

### 3.1 功能总览

| 编号 | 功能模块 | 简述 |
|------|----------|------|
| F-01 | PTP 客户端 | IEEE 1588 v2 从时钟侧：报文收发、BMC、偏移与路径延迟估计、状态机 |
| F-02 | 1588 ACR 客户端 | 在 PTP 时间戳与统计基础上，完成频率/相位跟踪、滤波与本地时钟恢复策略（含软件时间戳场景的误差与缓解说明） |
| F-03 | NTP 客户端 | NTP/SNTP 客户端：与 NTP Server 交互，维护本地时钟偏移与状态 |
| F-04 | 多客户端与多 Server | 多个独立客户端实例，各自绑定配置并与不同 Server/GM 交互 |
| F-05 | 配置管理 | 域号、接口、单播/组播、delay 机制、对端列表、实例标识等 |
| F-06 | 可观测性 | 日志、指标、关键状态对外查询（API 或等价方式） |

### 3.2 PTP 客户端（F-01）

**目标**：实现可运行的 **1588 PTP 从时钟**侧能力，能与标准 GM（如实验室设备、linuxptp）互通。

**主要能力（需在设计中细化，本文档为需求级）**：

- 支持 **PTP v2**；默认支持 **E2E** 或 **P2P** 之一为可配置项（或两者可配置，分阶段实现）。  
- 支持 **两步法** Sync / Follow_Up 与 **Delay_Req / Delay_Resp**（或与 P2P 对应的 Peer 延迟报文，依机制而定）。  
- 实现或集成 **BMC** 相关行为，完成主时钟选择与从时钟锁定状态管理。  
- 正确处理报文头中与时间相关的字段（含 **correctionField** 等与路径不对称相关的语义，以实现设计为准）。  
- 输出：**主从关系、offset、mean path delay（或等价量）、同步状态**（如锁定/失锁/告警）。

**约束**：未在配置或标准中确认的行为（如具体 G.8275.x 配置文件）不得默认实现为唯一路径，应通过配置开关与说明文档区分。

### 3.3 1588 ACR 客户端（F-02）

**目标**：在 PTP 客户端提供的测量与时间戳基础上，提供 **ACR** 路径：对本地时钟进行 **频率与相位跟踪**，并在抖动、丢包、不对称等条件下保持可接受的恢复性能。

**主要能力**：

- 基于 Sync/Follow_Up、Delay 等路径的样本，构建 **环路滤波 / PLL（或等价）** 输入。  
- 支持 **holdover** 策略（在报文中断或质量恶化时的行为需可配置并文档化）。  
- **软件时间戳**场景：明确 **误差模型** 与 **缓解手段**（批量补偿、滤波参数、采样策略等），与硬件时间戳路径区分配置与指标。  
- 输出：除 PTP 层指标外，ACR 层 **频偏估计、锁定质量、告警** 等（字段在设计阶段定稿）。

**与 F-01 关系**：ACR 逻辑依赖 PTP 客户端输出的时间戳与统计；宜通过清晰模块边界解耦，便于单独测试。

### 3.4 NTP 客户端（F-03）

**目标**：实现 **NTP（或 SNTP）客户端**，与 **NTP Server** 交互，维护本地时间与同步状态。

**主要能力**：

- 支持与一个或多个 **NTP Server** 通信（多 Server 策略：主备、轮询间隔、筛选算法等在详细设计中规定）。  
- 输出：**offset、delay、stratum、同步状态、当前选用的 Server** 等。  
- 与 PTP/ACR **并存**：多实例下可与 PTP 实例并行运行；是否允许同一进程内「单实例多协议」由实现决定，但需求上须支持「多客户端各自对接不同 Server」。

### 3.5 多客户端与多 Server 交互（F-04）

**目标**：同一部署单元内支持 **多个客户端实例**，每个实例独立与对端 **Server / GM** 交互，互不干扰（配置、状态、资源隔离）。

**需求要点**：

- 每个实例具备唯一 **实例 ID**（或等价标识），包含：协议类型（PTP / ACR 组合关系 / NTP）、绑定网络接口或本地地址、对端地址或组播参数、域号（PTP）等。  
- **资源隔离**：套接字、定时器、状态机、日志上下文按实例区分；避免单实例故障拖垮全局进程（详细设计给出隔离粒度与重启策略）。  
- **扩展性**：实例数量上限可配置，并受 **文件描述符、CPU、定时精度** 等环境约束；文档中给出推荐上限与压测方法。  
- **对端模型**：  
  - PTP：每个实例对应一组 GM/域/网络参数；  
  - NTP：每个实例对应一个或多个 NTP Server 及策略。

### 3.6 配置管理（F-05）

- 支持 **文件配置**（如 YAML/JSON/TOML），可选支持 **环境变量** 覆盖。  
- 配置项至少覆盖：实例列表、协议类型、网络参数、PTP 域与 delay 机制、NTP Server 列表与轮询参数、日志级别、指标开关。  
- **校验**：启动时对配置做 Schema 校验，错误信息可定位到实例与字段。

### 3.7 可观测性（F-06）

- **日志**：分级、可关联实例 ID；敏感信息（密钥若未来引入）脱敏。  
- **指标**：建议暴露 Prometheus 兼容或等价接口（可选），至少包括各实例 offset、delay、同步状态、报文速率错误计数。  
- **健康检查**：供编排系统探测进程与各实例关键状态（详细接口在接口设计中定义）。

## 4. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 在目标硬件与实例数量下，CPU/内存占用满足项目指标（待补充具体数字） |
| 可靠性 | 单实例异常可记录并可恢复；进程级崩溃行为与监管策略在运维文档中说明 |
| 安全性 | 绑定接口与最小权限（如 Linux `CAP_NET_RAW` 等）；仅监听必要端口；配置中的密钥安全存储（若适用） |
| 可维护性 | 模块化、类型标注、关键状态机有测试覆盖 |
| 兼容性 | 与主流 linuxptp / 标准 NTP Server 的互通在测试清单中验证 |

## 5. 系统架构需求（概要）

- **进程模型**：单进程多实例或多进程按实例拆分，二选一或组合需在概要设计中决策并说明运维影响。  
- **模块划分（逻辑）**：  
  - 网络 I/O 与报文编解码（PTP / NTP 分离或共享底层）  
  - PTP 状态机与 BMC  
  - ACR 滤波与调钟接口（与 OS 调时 API 的抽象层）  
  - NTP 客户端栈  
  - 配置中心、指标与日志  
- **对外 API**：是否提供 REST/gRPC 由后续迭代确定；本文档要求至少具备 **可查询的同步状态** 能力。

## 6. 接口需求（概要）

| 接口 | 方向 | 说明 |
|------|------|------|
| 网络 | 出站/入站 | PTP（组播/单播）、NTP，依实例配置 |

详细字段与 URL 在《接口设计说明书》中定义。

## 7. 假设与依赖

- 上游 **Server/GM** 符合所选标准与配置模式。  
- 网络路径上的 **不对称延迟** 可能影响 PTP 精度，需在部署指南中提示。  
- **硬件时间戳** 是否可用取决于网卡与驱动；不可用则走软件时间戳路径并启用 ACR/滤波相关配置。

## 8. 验收与测试要点

- **单元测试**：报文解析、状态迁移、滤波器数值样例。  
- **互通测试**：与 linuxptp GM、实验室 GM、公共/自建 NTP Server 的联调清单。  
- **多实例测试**：N 个实例同时运行，验证隔离与资源占用。  
- **验收标准**：各实例达到配置要求的同步状态，指标在约定阈值内持续稳定时间 T（由项目约定）。
